Why the tool-calling ≥50% bar was accepted, not hit. A tool turn is a full extra LLM call (message → requires_action, tool-result → final), so tool interactions are LLM-dominated, not transport-dominated — multi-tool is lower (24%), not higher, because more tool calls add more LLM time. The transport win itself is real and near-complete (~490 ms/turn, 96% of transport overhead removed); it simply can't be ≥50% as a fraction of multi-second LLM turns. The old stack was also measured warm — real first-use adds Lambda cold start, which would enlarge the reduction.

What was built

backend/ AgentCore container
The LangGraph-JS agent ported verbatim; only PostgresSaver → in-VM MemorySaver. New /invocations + /ping + /ws (streaming + tool round-trip collapse) + re-seed. ARM64, 325 MB.
client/ WebSocket transport
Fork of the DAVAI client with a session-pinned WS transport (8 unit tests), wired behind a default-off flag; threadId → runtimeSessionId; idle re-seed from the transcript.
infra/ IaC
Least-privilege IAM (no bedrock:InvokeModel — calls stay direct-to-provider), deploy runbook, and a one-block teardown.
done-loop/ Playwright + latency
The parity harness (real client in real CODAP) driving the 40/40 certification, plus the transport / old-vs-new latency tooling and results.

Reconcile plan → davai-plugin

  • Replace sam-server with the AgentCore container + IaC; add Google/Anthropic keys for full multi-provider parity (this spike wired OpenAI only).
  • Apply the client WS transport; set WS_SERVER_URL and enable the flag; keep the poll path for rollback.
  • The one piece not built (needed for production): a client-reachable auth path to the deployed runtime — AgentCore's invoke is a SigV4 AWS API, so the browser needs a thin SigV4 proxy or AgentCore OAuth inbound auth. Everything else is integration-ready.
